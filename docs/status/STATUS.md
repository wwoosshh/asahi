# 현재 상태

## 라이브 인프라

3-프로세스 하이브리드 구조로 상시 운영 중이다.

- **Railway 클라우드 봇** — 24/7 상시 구동, 디스코드 연결·대화 처리. 컨테이너는 무상태(stateless)이며
  재배포돼도 데이터가 사라지지 않는다. 절차는 `deploy/railway-셋업.md` 참고.
- **로컬 워커** — 소유자 PC에서 실행되며, 소유자 DM의 PC 작업(파일·Bash)을 봇으로부터 위임받아 처리한다.
  디스코드에는 연결하지 않고 DB 큐(`worker_jobs`)를 폴링한다.
- **Supabase Postgres** — 두 프로세스가 공유하는 정본 상태 저장소(유저·대화·기억·작업 큐). 로컬 PM2는
  Railway의 폴백/대체 수단이며, 봇 토큰이 1개라 둘을 동시에 띄우지 않는다(`deploy/PM2-명령어.md`).

## 병합된 주요 기능 (main)

- **멀티유저 런타임** — 대화별 직렬 처리·프라이버시 스코프(DM=사용자별, 서버=공유)·시간당 한도
  (손님: 유저별 20 / 전역 40, 소유자: 무제한)·크래시 복구.
- **원격 개발 워크플로우(Phase A)** — 소유자 DM에서 허용 폴더 안 파일·Bash 전권. 손님/서버는 PC 접근
  없음. 상세 위협모델은 `docs/security/capability-model.md` 참고.
- **캐릭터/페르소나** — 고정된 인격(이름·말투·소유자/손님/서버별 어조 차등)과 대화 이력 기반의 가벼운
  관계 진화. 예약어(`/새세션` 등)로 세션·톤을 초기화할 수 있다.
- **자기인지 DB 조회(조각C)** — 소유자 DM 전용 도구로 스키마 조회·읽기전용 SQL 질의·런타임 정보 제공.
  Postgres READ ONLY 트랜잭션 + 정적 SQL 가드 + 타임아웃으로 다층 방어한다.
- **디스코드 이미지 입력(멀티모달)** — 이미지 첨부를 모델에 직접 전달한다. 이미지 턴은 위임하지 않고
  봇이 직접 처리하며, 과거 이미지는 재주입하지 않고 마커만 저장한다(비용 방지).
- **모델**: 기본 Opus 계열 최신 모델로 고정, 환경변수로 재정의 가능.

## 테스트

`agent/` 기준 vitest 29개 파일, **319개 통과 + 1개 skip**. skip 1건은 Postgres READ ONLY 트랜잭션의
실제 쓰기 거부 동작으로, 테스트용 인메모리 Postgres 구현(pg-mem)의 한계상 유닛 테스트로 재현할 수 없어
실 Supabase 스모크로만 검증 가능하다.

## 미완 / 미검증

다음은 코드·유닛 테스트로는 검증됐지만, 실제 Supabase·실배포 환경에서의 스모크 테스트가 아직 없다.
체크리스트는 `deploy/smoke-test.md`에서 추적한다.

- Postgres READ ONLY 트랜잭션의 실제 쓰기 거부.
- 워커 ↔ 봇 실통신(작업 claim 원자성·폴링 주기·상태 배달, 서로 다른 머신 간 시계 차이 포함).
- 디스코드 이미지 첨부 → 모델 인식(실제 첨부 파일로 확인).
- `/새세션` 계열 명령으로 세션을 리셋했을 때 캐릭터 톤이 실제로 체감되는지.
- 손님·서버 채널에서의 한도·프라이버시 스코프 체감(유닛 테스트는 순수 로직만 커버).

## 알려진 한계

- **전송 유실 가능성**: 하드 크래시가 디스코드 응답 전송 도중 발생하면 그 응답 1건은 재전송되지 않는다
  (생성된 텍스트 자체는 이벤트 로그에 보존된다). 정상적인 그레이스풀 종료는 flush로 처리되어 유실이
  없다.
- **워커 위임은 소유자 전용**: 손님용 워커는 별도 인증·행 단위 권한 분리(RLS)가 아직 구현되지 않아
  지원하지 않는다. 워커를 운영하는 PC는 소유자 신원과 연결된 자격증명을 사용하므로, 그 자격증명이
  유출되면 위임 신뢰 경계가 무너진다 — 취급 정책은 `docs/security/capability-model.md`(및 승격 예정인
  `SECURITY.md`) 참고.
- **읽기전용 조회 결과 크기**: 자기인지 SQL 조회 도구는 현재 애플리케이션 단에서 결과를 자르는 방식이라,
  매우 큰 결과 집합에 대해서는 SQL 단 LIMIT 적용 등 후속 개선이 필요하다.
