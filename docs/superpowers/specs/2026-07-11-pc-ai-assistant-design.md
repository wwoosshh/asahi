# 개인 AI 비서 "상주 에이전트" 설계 문서

- **작성일**: 2026-07-11
- **상태**: 사용자 승인 완료 (아키텍처 A안 + 5단계 로드맵)
- **대상 환경**: Windows 11 데스크톱, 로컬 PC를 서버처럼 상시 구동

## 1. 개요와 목표

사용자의 PC에 상주하면서 사람 같은 매니저 역할을 하는 종합 AI 비서를 만든다.

- 디스코드 봇과 PC 전용 채팅 UI 양쪽에서 대화할 수 있다.
- 파일·셸·프로그램 실행, 브라우저 자동화, (보조적으로) GUI 조작까지 PC 작업을 대신 수행한다.
- 말을 걸면 답할 뿐 아니라, 상황을 감지해 먼저 연락하고, 장기 목표를 받아 자율적으로 수행한다.
- 세션이 바뀌고 PC가 재부팅되어도 **기억이 끊기지 않는다** — 기억은 로컬 저장소(SQLite + 마크다운)에 영구 보존된다.

## 2. 확정된 결정 사항

| 항목 | 결정 |
|---|---|
| LLM 백엔드 | Claude Pro/Max **구독 + Claude Agent SDK** (개인용 공식 지원). 한도 부족 시 API 키로 전환 가능 — 코드 변경 없이 인증만 교체 |
| 언어 | TypeScript / Node.js |
| 대화 채널 | 디스코드 봇 (1단계) + 로컬 웹 채팅 UI (4단계) |
| 능동성 | 반응형 대화 + 상황 감지 후 선제 연락 + 장기 목표 자율 수행 |
| PC 제어 범위 | 파일·셸·프로그램 실행(기본), 브라우저 자동화(Playwright MCP), GUI 조작(computer use — 5단계, 보조 수단) |
| 아키텍처 | **A안: 단일 상주 데몬 + Agent SDK 코어** (필요 시 추후 워커 분리로 진화) |
| 저장소 | SQLite(구조적 데이터) + 마크다운 폴더(큐레이션된 장기 기억) |
| 상시 구동 | PM2 데몬 + 부팅 자동 시작 + 절전 해제 |

### 명시적 제약

- 구독 인증은 **본인 개인용**에 한해 허용된다. 타인에게 서비스로 제공하려면 API 키로 전환해야 한다.
- 구독 사용량 한도(5시간 롤링 윈도우 + 주간 한도)를 소모하므로, 유휴 시 LLM을 호출하지 않는 **이벤트 기반 설계**가 필수다.
- PC가 꺼져 있으면 외부(디스코드)에서 불러도 응답할 수 없다 — 로컬 상주의 본질적 한계로 수용한다.

## 3. 아키텍처

Node.js 프로세스 하나가 PC에 상주하며, 세 층으로 구성된다.

```
[디스코드 어댑터]  [로컬 채팅 UI 어댑터]  [스케줄러/감지기]
        └──────────────┬──────────────┘
              이벤트 버스 (모든 입력을 통일된 이벤트로)
                        │
              에이전트 코어 (Agent SDK 세션 관리 + 기억 주입)
                        │
  [내장 도구: 파일/셸/웹]  [MCP: 브라우저]  [MCP: GUI]  [memory/ + SQLite]
```

설계 원칙:

- **어댑터**는 "메시지 받기/보내기"만 안다. **코어**는 "이벤트 처리"만 안다. **저장소**는 "기록"만 안다. 각 유닛은 이벤트 버스라는 좁은 인터페이스로만 통신한다.
- 디스코드 메시지, 채팅 UI 입력, 스케줄 타이머, 감지기 알림이 전부 동일한 이벤트 형태로 코어에 들어간다. 채널을 추가해도(예: 텔레그램) 코어는 변경되지 않는다.
- 코어는 이벤트를 큐로 직렬 처리한다(한 번에 하나의 세션). 긴 작업 중 도착한 메시지는 큐에 대기하며, 병렬성이 필요하면 Agent SDK 서브에이전트를 활용한다. 진짜 멀티 프로세스 병렬화(B안 요소)는 필요해질 때 도입한다.

## 4. 메모리 시스템 — "세션은 짧게, 기억은 영원히"

LLM 세션(컨텍스트)은 유한하므로 세션을 영원히 유지하는 방식은 불가능하다. 대신:

> 세션은 일회용이지만, 깨어날 때마다 로컬 저장소에서 기억을 재주입해 연속성을 만든다.

### ① SQLite (`store/agent.db`, better-sqlite3, WAL 모드)

| 테이블 | 내용 |
|---|---|
| `events` | 모든 대화·알림·작업 기록의 시간순 영구 로그 |
| `summaries` | 오래된 대화의 주기적 요약 (컨텍스트 재주입용) |
| `goals` | 장기 목표 정의와 진행 상태 |
| `schedules` | 예약 작업 정의 (크론식, 다음 실행 시각) |
| `settings` | 시스템 설정 (채널 ID, 한도 상한 등) |

- FTS5 전문 검색 인덱스를 `events`에 걸어 "지난달에 내가 뭐라고 했지?" 같은 회상 질의를 지원한다.

### ② 마크다운 메모리 폴더 (`memory/`)

- 비서가 스스로 큐레이션하는 장기 기억: 사용자에 대한 사실, 선호, 진행 중인 프로젝트, 배운 교훈.
- 파일 하나 = 기억 하나. `MEMORY.md`가 한 줄 요약 인덱스 역할.
- 사람이 직접 열어 확인·수정할 수 있다.

### 깨어날 때의 재주입 흐름

1. 이벤트 발생 → 코어가 Agent SDK 세션 시작
2. 시스템 프롬프트 + `MEMORY.md` 인덱스 + 최근 대화 요약(SQLite 조회)을 주입
3. 비서는 "어제 일을 기억하는 상태"로 응답
4. 세션 종료 시 새로 배운 것을 `memory/`에 기록하고 대화를 DB에 요약 저장

연속 대화 중에는 Agent SDK의 세션 이어가기(resume)를 사용해 같은 세션을 유지하고, 일정 시간 유휴 후에는 세션을 닫고 요약을 남긴다.

## 5. 상시 구동 — PC를 서버로

| 요소 | 방법 |
|---|---|
| 프로세스 상주 | PM2로 데몬 등록 — 크래시 시 자동 재시작, 로그 로테이션 |
| 부팅 자동 시작 | `pm2-windows-startup` — Windows 업데이트 재부팅 후에도 자동 복귀 |
| 절전 방지 | Windows 전원 옵션에서 절전/최대 절전 해제, 모니터만 끄기 |
| 크래시 복구 | 모든 상태가 SQLite에 있으므로 재시작 후 이어서 동작. 처리 중이던 이벤트는 미완료 플래그로 재시도 |
| 디스코드 재연결 | discord.js 내장 자동 재연결 |
| 자체 점검 | 코어가 주기적으로 헬스 상태(가동 시간, 큐 길이, 마지막 오류)를 DB에 기록 |

## 6. 컴포넌트 구성

```
agent/
├── src/
│   ├── core/          # 에이전트 코어: 세션 생성·큐 관리·기억 주입·응답 라우팅
│   ├── adapters/
│   │   ├── discord/   # discord.js 봇 (DM + 지정 채널)
│   │   └── chat-ui/   # Fastify + WebSocket 로컬 웹 채팅 (4단계)
│   ├── events/        # 이벤트 버스와 이벤트 타입 정의
│   ├── scheduler/     # croner 기반 예약 실행 + 감지기 모듈 (3단계)
│   ├── store/         # SQLite 접근 계층 (better-sqlite3)
│   └── goals/         # 장기 목표 엔진 (5단계)
├── memory/            # 마크다운 장기 기억
├── mcp/               # MCP 서버 설정 (Playwright 브라우저, GUI 조작)
└── docs/
```

각 유닛의 책임과 인터페이스:

| 유닛 | 하는 일 | 인터페이스 | 의존 |
|---|---|---|---|
| 이벤트 버스 | 이벤트 발행/구독 | `publish(event)`, `subscribe(type, handler)` | 없음 |
| 코어 | 이벤트 → 세션 → 응답 | 이벤트 버스 구독/발행 | Agent SDK, store, memory |
| 디스코드 어댑터 | 메시지 수신/발신 | 이벤트 버스 | discord.js |
| 스케줄러 | 시각 도달 시 이벤트 발행 | 이벤트 버스 | store(schedules) |
| store | 영구 기록/조회 | 타입된 쿼리 함수 | better-sqlite3 |

## 7. 데이터 흐름 (대표 시나리오)

**반응형 (디스코드에서 말을 걸었을 때):**

1. 디스코드 어댑터가 메시지 수신 → `user_message` 이벤트 발행
2. 코어가 이벤트를 큐에 넣고 SQLite `events`에 기록
3. 코어가 Agent SDK 세션 시작(또는 진행 중 세션에 이어붙임) — 기억 재주입 포함
4. 비서가 도구를 사용하며 작업 수행 (중간 진행 상황도 이벤트로 발행 가능)
5. 응답 이벤트 발행 → 디스코드 어댑터가 전송, SQLite에 기록
6. 유휴 타임아웃 후 세션 정리: 기억 기록 + 요약 저장

**능동형 (스케줄/감지):** 출발점이 스케줄러 타이머 또는 감지기라는 점만 다르고 동일한 흐름. 응답은 기본 채널(디스코드 DM)로 전달.

**장기 목표 (5단계):** `goals` 테이블의 활성 목표를 스케줄러가 주기적으로 깨움 → 코어가 "목표 진행" 세션 실행 → 진행 상황을 `goals`에 갱신, 중간보고를 디스코드로 발송.

## 8. DB 관리·점검 플로우 (SQL 비숙련 사용자용)

- **1차 수단 — 비서에게 묻기**: "지난주 대화 보여줘", "스케줄 목록 정리해줘" 등 자연어로 질의하면 비서가 DB를 조회해 표로 답한다. SQL은 비서가 대신 작성한다.
- **2차 수단 — GUI 도구로 직접 검증**: **Beekeeper Studio** (사용자 선택 도구, 무료 Community 버전). SQLite 연결로 `E:\Asahi\store\agent.db`를 열어 표 형태로 탐색·필터·정렬. Read Only Mode는 유료 기능이므로, 무료 버전에서는 **조회(SELECT)만 하고 셀을 직접 수정하지 않는 규율**로 대체한다 — WAL 모드에서 읽기는 데이터를 변경하지 않으므로 안전하다. 완전히 무위험으로 보고 싶으면 라이브 파일 대신 **백업 사본**(`store/backups/`)을 연결한다.
- DB는 단일 파일(`E:\Asahi\store\agent.db`)이므로 백업은 파일 복사로 충분하다. 비서가 주 1회 자동 백업(`store/backups/`)을 수행한다.

## 9. 안전장치

- **승인 게이트**: 파일 삭제·외부 전송·구매 등 되돌리기 어려운 작업은 Agent SDK 권한 훅으로 가로채 디스코드로 승인을 요청하고, 허용 응답이 있을 때만 실행한다.
- **GUI 조작 제한**: computer use는 사용자가 명시적으로 요청한 작업에만 사용한다.
- **한도 보호**: 시간당 세션 수 상한을 설정에 두고, 구독 한도 근접이 감지되면 비서가 스스로 알리고 능동 작업을 보류한다.
- **채널 인증**: 디스코드 봇은 지정된 사용자 ID(소유자)의 메시지에만 반응한다.
- **비밀 관리**: 디스코드 토큰 등은 `.env`로 관리하고 저장소에 커밋하지 않는다.

## 10. 에러 처리

- 어댑터 오류(디스코드 끊김 등)는 자동 재연결로 복구하고 이벤트 버스에 상태 이벤트를 남긴다.
- 세션 오류(LLM 호출 실패, 한도 초과)는 지수 백오프로 재시도하고, 반복 실패 시 사용자에게 오류 사실을 그대로 보고한다.
- 처리 중 크래시가 나도 이벤트가 DB에 미완료 상태로 남아 재시작 시 재처리된다.
- 모든 오류는 PM2 로그 + DB `events`에 기록된다.

## 11. 테스트 전략

- **단위 테스트 (vitest)**: store 계층(스키마·쿼리), 이벤트 버스, 스케줄러 계산 로직. LLM 호출은 모킹.
- **통합 스모크 테스트**: 각 단계 완료 시 실제 플로우를 수동 검증하는 체크리스트 (예: 1단계 — 디스코드 DM → 응답 수신 → 재시작 → 기억 유지 확인).
- 어댑터·MCP처럼 외부 의존이 큰 부분은 통합 테스트 위주로 하고 과도한 모킹은 피한다.

## 12. 단계별 로드맵

각 단계는 독립된 구현 계획 → 구현 → 검증 사이클로 진행한다.

| 단계 | 내용 | 완성 시 가능한 것 |
|---|---|---|
| 1 | 코어 + SQLite/메모리 + 디스코드 봇 + PM2 상시구동 | 기억이 이어지는 비서와 디스코드로 대화 |
| 2 | 파일/셸 도구 활성화 + 브라우저 자동화 MCP | "다운로드 폴더 정리해줘", "검색해서 요약해줘" |
| 3 | 스케줄러 + 상황 감지기 | 매일 아침 브리핑, 중요 상황 감지 시 선제 연락 |
| 4 | 로컬 웹 채팅 UI | PC 전용 창으로 대화 |
| 5 | 장기 목표 엔진 + GUI 조작(computer use) | "이번 주에 X 조사해줘" 자율 수행, 임의 앱 조작 |

### 단계 시작 시 확정할 결정 포인트

- **3단계**: 어떤 감지기를 넣을지(파일 변화, 특정 사이트, 시스템 상태 등)는 3단계 계획 시 사용자와 목록 확정.
- **5단계**: Windows GUI 입력 제어 라이브러리(후보: nut.js 계열, Windows API 기반 커스텀 MCP)는 5단계 계획 시 평가 후 확정.
