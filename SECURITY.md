# 보안 정책

Asahi는 소유자 개인용 비서로, 디스코드를 통해 대화하며 소유자 PC 에서 소유자 권한으로
파일·셸 작업을 수행할 수 있다. 이 문서는 위협 모델을 공개용으로 요약한다. 상세 능력 계층과
알려진 한계는 `docs/security/capability-model.md`, `docs/security/risk-register.md` 를 참고한다.

## 자산

- 소유자 PC 에 대한 파일·셸 접근 권한(원격 개발 워크플로우).
- 정본 상태 저장소(Postgres/Supabase) — 대화 이력, 기억, 사용자 권한, 작업 큐.
- 디스코드 봇 계정 및 소유자와의 1:1 DM 채널.

## 주요 행위자·위협

- **손님 사용자** — 소유자가 아닌 디스코드 사용자. 대화·기억(본인 스코프)만 가능해야 하며,
  PC 도구·전원 기억 조회·DB 조회 같은 특권에 접근할 수 없어야 한다.
- **관찰 콘텐츠를 경유한 프롬프트 인젝션** — 대화 중 모델이 읽는 텍스트(첨부 이미지 포함)에
  숨겨진 지시가 실려 올 수 있다. 서버/스레드 채널은 참여자가 늘거나 이력을 누구나 볼 수 있어
  사설 1:1 을 보장하지 못하므로, 신뢰 경계를 별도로 둔다.
- **클라우드 컨테이너 실행 환경** — 상시 구동되는 클라우드 봇은 소유자 PC 가 물리적으로 없는
  컨테이너다. 이 환경에서는 PC 도구 자체가 열리지 않아야 한다.
- **유출된 DB 연결 문자열** — 소유자와 로컬 워커가 공유하는 비밀이 유출되면 위임 신뢰 경계가
  흔들릴 수 있다.

## 완화책 요약

| 위협 | 완화 |
| --- | --- |
| 손님이 PC/특권 도구에 접근 시도 | 신원(`isOwner`) 기반 게이팅 — 역할(role) 을 바꿔도 신원은 바뀌지 않는다. `docs/security/capability-model.md` §신원 vs 역할 게이팅 |
| 프롬프트 인젝션으로 허용 폴더 밖 파일 접근 | 모든 파일/Bash 호출을 `canUseTool` 이 실제 경로 기준으로 재검증(realpath 정규화 + 허용 폴더 대조). `docs/security/capability-model.md` §경로 게이팅 |
| 클라우드 컨테이너에서 PC 작업 시도 | 도구셋 계산 단계와 `canUseTool` 재검증 단계, 두 곳에서 각각 거부(이중방어) |
| DB 연결 문자열 유출 | 워커 위임은 소유자 신원일 때만 발생하도록 정책으로 제한. 상세는 `docs/security/risk-register.md` §1–2 |
| 임의 SQL 실행 시도 | 애플리케이션 사전검사 + Postgres READ ONLY 트랜잭션의 이중 방어. `docs/security/capability-model.md` §READ ONLY SQL 가드 |
